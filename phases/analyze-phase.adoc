= ONTAP Foreign LUN Import（FLI）分析フェーズ
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
分析フェーズでは、詳細な移行計画と実行に進む前に修正すべき事項を特定します。このフェーズの主な成果は、すべてのホストと、それに依存する完全な SAN スタック（OS、ボリューム マネージャ、HBA、マルチパス、スイッチファームウェア、ターゲット ONTAP リリース）を、サポートされている移行後構成に取り込めるようにすることです。これは、各ホストの現在の状態を NetApp Interoperability Matrix（IMT）でサポートされている組み合わせと比較し、ホストごとに明確なターゲット構成（OS レベル、MPIO/ALUA 動作、HBA ドライバ/ファームウェア、および必要なホスト ツール）を定義し、その目標を達成するために必要な具体的な変更点を一覧にしたギャップ分析を作成することによって行われます。

アプリケーションの互換性や停止計画に影響を与える可能性のある修正変更を避けるため、ギャップ分析をスケジュールしてレビューします。多くの環境では、OSのパッチ適用やHBAのアップデートなど、一部の変更は移行期間前に完了させておく方が安全です。しかし、マルチパススタックの変更は、MPIOの動作を変更すると、現在のアレイとのサポート性に影響を与え、ロールバックを複雑にする可能性があるため、多くの場合、切り替え時または移行後に延期されます。

iSCSI FLIの場合、分析フェーズではiSCSI固有の準備状況も把握する必要があります。

* 切り替え後もホストのiSCSIマルチパス設計が引き続きサポートされていることを確認してください
* 外部アレイがiSCSIバックエンドでの使用に適していることを確認してください。バックエンドのトランスポートの違いによって、相互運用性の動作が変わる可能性があるためです。




== ONTAP FLI分析フェーズのタスク

分析フェーズはホスト中心ですが、完全なエンドツーエンドサポートチェーン（ホストからファブリックからONTAPへ、そして再び戻る）を検証し、FLIバックエンドの相互運用性の位置を確認する必要があります。

[cols="1,2"]
|===
| コンポーネント | タスク 


| ホスト  a| 
* 現在のホスト構成をIMTでサポートされている構成と比較し、移行後のターゲット構成を決定します。
+
** OSレベル
** マルチパス アプローチ
** HBAモデル、ドライバ、ファームウェア
** 必要なホストツール


* 必要な項目を含むホストごとのギャップ分析を作成します：
+
** ホットフィックスとパッチ
** OSの更新
** HBAドライバおよびファームウェアのアップグレード
** マルチパスの変更
** 必要なユーティリティをすべてリストアップしてください。






| ファブリック（FC / iSCSIネットワーク）  a| 
* エンドツーエンドビューでスイッチとファブリックのサポート（モデルとファームウェア）を検証し、ターゲットONTAPとホストの組み合わせでファブリックがサポートされていない場合は、必要なアップグレードを記録します。




| NetAppストレージアレイ（デスティネーション）  a| 
* ターゲットのONTAPプラットフォームとリリースが、該当するAFF有効化の期待事項を含め、意図されたFLIワークフローと互換性があることを確認し、ターゲット設計に影響を与える前提条件をすべて記録します。




| 外部ストレージアレイ（ソース）  a| 
* ソースアレイがFLIバックエンドの相互運用性に対応しているかどうかを検証します。ONTAP 9.9.1以降のリリースでIMTに相互運用性がリストされていない場合は、SAN LUN Migrateアプリを使用して、サポートされているかどうかを判断するのに役立ちます。


|===


== ONTAP FLI分析フェーズでIMTを使用する

Interoperability Matrix Tool（IMT）は、構成のリポジトリであり、NetApp製品がサードパーティ製コンポーネントと相互運用するサポート対象のエンドツーエンドの結果構成に絞り込むためのガイド付き方法です。IMTには、サポート対象構成と認定済み構成の両方を含めることができます：

* サポートされている構成は、NetAppによって認定されています。
* 認定された構成は、NetAppコンポーネントで動作することが第三者によって確認されています。


.IMTで次の項目を確認します。
* ワークシートにIMTアクションを記録：計画ワークシートに、ホストとスイッチの両方についてIMTが推奨するソフトウェア、パッチ、およびアップグレードを記載してください。
* 最初は広く始め、次に絞り込む：静的な基準（ONTAPリリース、プロトコル、HA/クラスタモード）から始め、次にサイト調査を参考にホストOS、ボリューム マネージャ、HBAの詳細を追加します。
* 過剰なフィルタリングは避けてください：検索条件が細かすぎて結果が表示されない場合は、フィルタを緩め、複数の有効な設定を確認し、最もサポートされている条件を選択してください。
* HBA識別子を正規化します：HBAはOEM部品番号として表示される場合があります。IMTに入力する前に相互参照してください。
* すべてのホストをチェックします：移行対象範囲に含まれる各ホストについて、IMTサポート可能性チェックを実施する。
* 相互運用性チェックを2つ実施してください：
+
** バックエンドFLIサポート：ソースアレイのバックエンドシナリオがサポートされていることを確認してください。
** エンドツーエンドの移行後サポート：ONTAP、ホストOS、マルチパス、HBA、およびスイッチファブリックの組み合わせに対するサポートを確認します。


* 実用的なIMTワークフロー：
+
** ソースアレイモデルを検索し、「Foreign LUN Import (FLI) backend Interoperability」を選択して、ONTAPプラットフォームとONTAPリリースを選択します。
** ホストの互換性については、「ONTAP SANホストを使用したエンドツーエンドビューの構築」を使用して、OS、マルチパス、およびHBAを検証してください。
** スイッチの互換性については、SANホストビューから「Build end-to-end view for SAN-Switch」を使用して、ファブリックとスイッチのサポートを検証してください。




.外部アレイがIMTに記載されていない場合（ONTAP 9.9.1以降）
ONTAP 9.9.1以降、外部アレイがIMTでサポート対象としてリストされていない場合は、SAN LUN Migrate Toolを使用して、そのアレイがFLIでサポートされる可能性があるかどうかを判断できます。このアプリはNetApp Support Siteからダウンロードし、ソースアレイLUNへのブロックアクセス（FCまたはiSCSI）が可能なLinuxホストから実行します。結果はエンジニアリング部門によって検証されます。外部アレイからのインポートは、現場で検証できます。iSCSI FLIの場合、外部アレイからのインポートは常に現場で検証されます。



== ONTAP FLIホストの変更とロールバック戦略

分析フェーズでは、変更内容と変更時期を明確に決定する必要があります。なぜなら、ホストの変更はリスクをもたらす可能性があり、既存のストレージのサポートにも影響を与える可能性があるからです。ホストスタックがONTAPでサポートされる前にアップグレードが必要な場合、これは重要です。

* ホストのアップデート（パッチ、HBAファームウェア、ドライバ）は、可能な限り事前に実施することで安全性が高まりますが、それでもメンテナンス期間やアプリケーションチームとの連携が必要です。
* マルチパススタック（MPIO、ALUA、およびサードパーティ製マルチパス）への変更は、現在のアレイのサポート可能性に影響を与える可能性があるため、移行イベントまで、あるいは切り替え後まで延期されることがよくあります。
* MPIOの再構成を遅らせることで、障害発生期間中に元のストレージに戻す必要が生じた場合に、ロールバックが容易になります。




== ONTAP FLI ギャップ分析

ギャップ分析では、現在の環境とNetApp推奨のターゲット構成を比較し、サポート対象の移行後状態を実現するために必要なアップグレードまたは修復事項の一覧を示します。

ホストごとの完全なターゲット構成には、通常、次のものが含まれます：

* OS構成レベルおよび必要なパッチまたはホットフィックス
* マルチパス方式と必要な設定（MPIOとALUAの動作
* HBAおよびCNAのモデル、ドライバ、ファームウェアのバージョン
* 該当する場合、お使いの環境に関連するホストユーティリティおよびツールの要件


ギャップ分析は、修復措置にダウンタイム ウィンドウが必要になる可能性があり、アプリケーション サポートの要件と相互作用する可能性があるため、速やかに確認する必要があります。
