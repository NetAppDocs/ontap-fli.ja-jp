= ONTAP FLI Linuxホストの修復
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
Linux ホストで ONTAP FLI 移行を実行する前に、以下の移行前手順を完了してください。オフライン移行とオンライン移行には、それぞれ別の手順が必要です。



[[pre-migration-offline-fli]]
== 移行前 - オフラインFLI

これらの手順は、オンラインFLIとオフラインFLIの両方に共通です。ホストからソースLUNのプロビジョニングを解除する前に、以下の手順が完了していることを確認してください。

[cols="3"]
|===
| ファイルシステム（LVMおよび非LVM） | LVM搭載Pacemaker（SCSI3 PR） | SANBOOT 


| ステップ1：エンドツーエンドのブロックデバイス設定を収集：マルチパスの健全性/パス、ブロックデバイス上のLVMレイアウト、および正しいブートデバイスとブートパラメータを確認するGRUB/ブートローダーの設定。 | ステップ1：エンドツーエンドのブロックデバイス設定を収集：マルチパスの健全性/パス、ブロックデバイス上のLVMレイアウト、および正しいブートデバイスとブートパラメータを確認するGRUB/ブートローダーの設定。 | ステップ1：エンドツーエンドSANブート構成を収集する：マルチパスの健全性/パス、ブートLUN上のLVMレイアウト、および正しいブートデバイスとブートパラメータを確認するGRUB/ブートローダー設定。 


| ステップ2：各データデバイス（LVM/非LVM）の `/etc/fstab`エントリがFS UUIDで参照されていることを確認します。 | ステップ2：移行対象のLUNに対応するクラスタリソースグループを無効にします。無効化したら、リソースグループを削除します。 | 手順2：データデバイス（lvm/nonlvm）を含むSANブートデバイスの `/etc/fstab`エントリが、FS UUIDで参照されていることを確認します。 


| ステップ3：ターゲットマウントポイントへのすべての操作が一時停止されていることを確認します。ブロックデバイスにアクセスしているプロセスが残っていないことを確認してください。必要に応じて、開いているプロセスハンドルをドレイン/強制終了します。 | ステップ3：先ほど無効化/削除した各リソースグループについて、LVM VG構成のバックアップをファイルに実行します。続いてVGの非アクティブ化が行われます。 | ステップ 3： `GRUB_CMDLINE_LINUX`行が正しいUUID参照で更新されていることを確認します。UUIDデバイス参照に基づいて、新しいGRUB設定を再作成します。 


| ステップ4：ファイルシステムをアンマウントし、LVMボリュームグループ構成のバックアップを実行します。VGを非アクティブ化し、VGをエクスポートします | ステップ4：PCS stonith設定からソースLUNを削除します。 | ステップ4：新しいgrub設定で変更を検証し、SANブートデバイスの変更を反映させるために新しいinitrdイメージを再作成します。 


| ステップ5：デバイスのマルチパス構成をフラッシュし、すべての古いSCSIデバイスファイルを削除します。 | ステップ5：デバイスのマルチパス構成をフラッシュし、すべての古いSCSIデバイスファイルを削除します。注：LUNの永続的な予約が見つかった場合は、それらをクリアし、PRキーがないことを確認してください。 | ステップ5：ホストをリブートして、上記の変更でホストが正常に起動できることを確認します。ホストが正常に起動したら、FLI移行のためのホスト事前修復の準備が整ったことを確認します。 


| ステップ6：マルチパス構成を再読み込みして、ホストに最新のデバイスリストが反映されるようにします。必要に応じて、ホストSCSIバスの再スキャンを実行してください。 | ステップ6：ホストを再スキャンして、既存のデバイスマッピングに古いエントリがないことを確認します。 | ステップ 6：次に sanboot ホストをシャットダウンして、ONTAP FLI ワークフローを続行します。 
|===


[[pre-migration-online-fli]]
== 移行前 - オンラインFLI

以下にオンラインFLIの具体的な手順を示しますが、これらの手順は移行前のオフラインFLIの手順を完了してから実行する必要があります。以下の手順は、NetApp LUNがホストにプロビジョニングされた後に実行する必要があります。これらの手順は、NetApp LUNがホストにプロビジョニングされた後に実行する必要があります。また、これはオンラインFLIにのみ適用されます。

[cols="3"]
|===
| ファイルシステム（LVMおよび非LVM） | LVM搭載Pacemaker（SCSI3 PR） | SANBOOT 


| ステップ1：マッピングされたすべてのLinuxホストでストレージを再スキャンします。 | ステップ1：マッピングされたすべてのLinuxホストでストレージを再スキャンします。 | ステップ1：サーバーの電源をオンにします。 


| ステップ2：マルチパスデバイスを検出する。 | ステップ2：マルチパスデバイスを検出する。移行したLUNがPCS Stonith構成に追加されていることを確認してください。 | ステップ2：サーバーのHBA BIOSまたはUEFIで、このホストHBAに提示される新しいブートデバイスを設定します。 


| ステップ3：ボリュームグループをインポートし、VGのアクティベーションを実行します。 | ステップ3：新しく検出されたVG/LVMに対してVG構成バックアップの復元を実行し、その後VGをアクティブ化します。 | ステップ3：ホストを再起動し、LUNインポートの一部として新しくマッピングされたLUNから起動します。 


| ステップ4：LVM構成をそれぞれのマウントポイントにマウントします。 | ステップ4：新しいPCSリソースグループを作成し、利用可能なLVとファイルシステムに対応するリソースを追加します。注：新しくアクティブ化されたLVMオブジェクトは、LVとファイルシステム構造をそのまま保持していることにご注意ください。 |  


|  | ステップ5：すべてのリソースがオンラインで利用可能なステータスであることを、PCSクラスタのステータスで確認します。 |  
|===
