= ONTAP Foreign LUN Import（FLI）の計画と準備フェーズ
:allow-uri-read: 
:icons: font
:imagesdir: ../media/


[role="lead"]
計画および準備フェーズでは、発見および分析の結果を実行可能な移行計画に変換し、移行期間に向けて環境が準備できていることを確認します。プロジェクトの作業の大部分は通常この段階に集中します。なぜなら、ここでオンラインまたはオフラインでの切り替え方法を最終決定し、エンドツーエンドの接続性を検証し、本番環境に移行する前に1つ以上のテスト移行を通じてワークフローを証明するからです。

iSCSI FLIの場合、このフェーズではiSCSIバックエンドの設計上の制約も検証する必要があります。ONTAPのiSCSIソフトウェアイニシエータはIC LIFを使用し、明示的なターゲットポータルIPとターゲットIQNを必要とし、検出セッションは不要で、定義されたスケール制限（ノードあたりのインポート数、LIF/ノードあたりのセッション数、LIFあたりのイニシエータ数、最大インポートサイズ）内で動作するように設計されています。



[[ontap-fli-plan-and-prepare-phase-results]]
== ONTAP FLI計画および準備フェーズの結果

* ホストギャップ分析を使用して修復計画を最終決定します。これには、移行前に実施される変更や、MPIOの変更など、切り替え期間中に実施される変更が含まれます。
* ソースから宛先へのLUNマッピング、ソースLUN識別子、およびシリアル番号を宛先のONTAP LUN名とパス（ロールバックマッピングを含む）に構築します。
* IC LIF到達性と、各HAノードが各ターゲットポータルとのセッションを確立するフルメッシュセッショントポロジーを使用して、ONTAPから外部アレイへのバックエンド接続を検証します。これは、信頼性の高い外部LUN検出の前提条件です。
* 構成の正確性を確認し、本番稼働期間と必要な人員を推定するのに役立つスループットサンプルを収集するために、1回以上のテスト移行を実行します。




[[ontap-fli-plan-and-prepare-phase-tasks]]
== ONTAP FLIの計画と準備フェーズのタスク

以下は、各コンポーネントのONTAP FLI計画および準備タスクです。

[cols="1,2"]
|===
| コンポーネント | タスク 


| ホスト  a| 
* ギャップ分析を使用してホスト修復計画を最終決定します。これには、変更を移行前、カットオーバー時、または移行後のいずれに実施すべきかを含めます。
* ダウンタイム許容度とワークロードの重要度に基づいて、オンラインまたはオフラインの移行モードを選択し、それに応じて修復のタイミングを調整してください。
* 移行期間中に使用する、想定されるユースケース（標準LUN、クラスタディスク、RDM/VMFSデータストアなど）におけるホスト側の再検出および修復手順を検証し、文書化する。




| ファブリック／ネットワーク  a| 
* FC環境：既存のゾーンセットをバックアップし、カットオーバーおよび移行後のクリーンアップに必要なゾーン変更を計画します。
* iSCSIバックエンド環境：ONTAP IC LIFと外部アレイのターゲットポータルIP間のエンドツーエンドのIP到達可能性を検証し（ルーティング/VLAN/ファイアウォール、TCP/3260を含む）、冗長性を確保します。
* 既存のIC LIFが既に別のサブネットで使用されている場合、外部ターゲットに到達するためだけに、別のサブネットに新しいIC LIFを追加しないでください（サポートされていない計画として扱ってください）。




| NetApp ストレージアレイ（デスティネーションONTAP）  a| 
* iSCSI FLI（サポートされているリリース/ECV）に対するONTAPクラスタの準備状況を確認し、移行専用のiSCSIバックエンド構成が移行後に削除されること（アップグレード/リバートの準備状況にとって重要）を確認します。
* LUNを所有するノードとHAパートナー上のIC LIFを特定して確認し、回復力（通常はノードごとに異なるポートに2つのIC LIF）を計画します。
* ソフトウェアiSCSIイニシエータ（IC LIFごとに1つのイニシエータ）を作成し、フルメッシュセッション（各HAノードが各外部ターゲットポータルおよびIQNに対応）を確立し、セッションが正常であることを確認します。
* storage disk showコマンドを使用してONTAPで外部LUNを検出し、512B要件を含むパスの健全性と基本プロパティを検証してから、外部LUNを外部としてマークします。
* -foreign-disk パラメータを使用して、サイズが一致するテスト用宛先 LUN をプロビジョニングし、計画されているフロントエンド プロトコル（FC または iSCSI）を使用してマッピングし、少なくとも 1 つのテスト移行を実行して、エンドツーエンドのワークフローを検証し、スループットをキャプチャします。
* プランがiSCSI FLIの制約（検出セッションなし、MCSなし、ERL0のみ、インポートとセッションの制限、最大インポートサイズ）を遵守していることを確認してください。




| 外部ストレージアレイ（ソース）  a| 
* ターゲットポータルのIPアドレスとターゲットIQNを収集します。セッションは明示的に作成され、検出セッションは使用されません。
* ONTAPイニシエータ用のベンダー側アクセスオブジェクト（ホスト/ホストグループ相当）を作成し、認定とテストのためにONTAPイニシエータのみに小さなテストLUN（通常は約1GB）をマッピングします。
* 外部ターゲットがONTAPイニシエータのログインとセッションを認識していること、およびインポート用に正しいLUNがマッピングされていることを確認します。
* 切り替え時に、本番ホストに同じソースLUNが同時に提示されないようにして、二重提示を回避してください。
* 本番環境へのインポート前に、サポートされている認定ツールと手法を使用して、外部アレイの認定を完了してください。


|===
